## 상황 설명

> 당신은 팀 프로젝트의 백엔드 개발자입니다. 간단한 웹 앱의 `main` 브랜치에 새로운 기능을 추가하고 푸시했는데, 그 커밋에 **치명적인 버그**가 포함되어 있었습니다. 팀원들은 이미 `git pull`을 받아 작업 중입니다.

이 시나리오를 선택한 이유: 실무에서 **가장 빈번하게 발생하는 실수**이며, 대응 방법을 잘못 선택하면 팀 전체의 Git 히스토리가 꼬일 수 있기 때문입니다.

## 학습 목표

1. `git revert`로 이력을 보존하면서 안전하게 커밋을 취소하는 방법을 배운다
2. `git reset`과 `git revert`의 차이를 명확히 이해한다
3. 공유 브랜치(main)와 개인 브랜치에서 각각 어떤 전략을 써야 하는지 판단할 수 있다
4. `git push -f`의 위험성과 올바른 사용 상황을 이해한다
5. `git log`와 `git diff`로 변경 사항을 추적하는 습관을 기른다

## 사전 요구사항

- Git 기본 명령어 (`init`, `add`, `commit`, `push`, `pull`) 사용 경험

- 터미널 기본 조작 (파일 편집, 디렉토리 이동)

## 핵심 학습 개념

### 왜 `revert`가 필요한가?

실수로 잘못된 코드를 푸시했을 때, 두 가지 선택지가 있습니다:

**선택지 1: `git reset` — 히스토리를 되감기**

- 커밋 자체를 히스토리에서 제거
- 혼자 쓰는 브랜치에서는 OK
- **공유 브랜치에서 사용하면 재앙** (다른 사람의 히스토리와 충돌)

**선택지 2: `git revert` — 취소 커밋을 새로 만들기**

- 잘못된 커밋을 되돌리는 **새로운 커밋**을 생성
- 히스토리가 보존되므로 **팀원에게 영향 없음**
- 공유 브랜치에서의 **안전한 표준 방법**

### Reset vs Revert 동작 비교

```
[상황] A --- B --- C(버그) 가 main에 push된 상태

─── git reset (히스토리 제거) ───
로컬:  A --- B                    (C가 사라짐)
원격:  A --- B --- C              (여전히 C가 있음)

이 상태에서 그냥 git push를 하면 GitHub가 거절합니다.
이유: "GitHub에는 이미 C가 있는데, 네 로컬에는 왜 C가 없니? 역사가 꼬였으니 먼저 Pull 받아서 합쳐!" (Non-fast-forward error)
→ push -f로 원격도 덮어써야 함 → 팀원 히스토리 꼬임!

`git push -f` 실행
GitHub의 경고를 무시하고 **"내가 맞으니까 내 로컬 기록으로 GitHub를 바꿔버려!"**라고 강제로 밀어넣는 작업
결과: GitHub에 있던 커밋 C가 삭제됩니다.
로컬: A → B (HEAD)
GitHub: A → B (main) <-- 이제 원격에서도 C는 사라짐

🚨 이게 왜 위험할까요? (협업 시 대참사 발생)
만약 **동료 '철수'**가 이미 커밋 C를 자신의 컴퓨터로 받아놓았다면 어떻게 될까요?

1. 철수의 컴퓨터: A → B → C
2. GitHub (당신이 바꾼 상태): A → B
3. 철수가 다음 날 git pull을 받으면?
  - GitHub에는 C가 없는데 철수한테는 C가 남아있습니다.
  - 철수의 Git은 혼란에 빠집니다. "어? 내가 가진 C를 GitHub에 올려야 하나? 아니면 GitHub에서 C가 지워진 건가?"


─── git revert (취소 커밋 생성) ───
로컬:  A --- B --- C --- C'       (C를 되돌리는 C' 생성)
원격:  A --- B --- C --- C'       (pull만 하면 자동 반영)
→ 팀원은 그냥 pull 받으면 됨 → 안전!

C를 되돌리는 C' 커밋 ??
- 코드의 변화(Diff)를 거꾸로 적용한다는 뜻
- git revert C를 실행하면, Git은 자동으로 C와 정반대의 작업을 하는 **새로운 커밋 C'**를 만듬.
- 이렇게 하면 C 커밋의 기록은 여전히 남으면서도(우리가 무슨 실수를 했는지 알 수 있음), 현재 상태는 C를 하기 전인 B 상태와 똑같아지게 됨.
- C가 B를 더럽혔다면, C'는 그 더러운 부분을 다시 닦아냈기 때문에 **파일의 내용물(Snapshot)**은 B 시점과 100% 똑같아지고,
revert로 만든 C'는 B와 내용은 같지만, **"지금 이 순간(시간)"**에 만들어졌고, **"C 다음(부모)"**에 온 자식이기 때문에 B와는 전혀 다른 고유한 해시값을 가지게 됨.
```

## 추천 실습 순서

1. **미션 1**: 현재 상태 파악 (git log, git diff로 버그 확인)
2. **미션 2**: `git revert`로 안전하게 되돌리기 (핵심!)
3. **미션 3**: revert 결과를 원격에 push
4. **미션 4**: (심화) 개인 브랜치에서 `reset + force push` 체험

## 교육자의 팁

- `git revert`는 **새로운 커밋을 만드는 것**입니다. "되돌리기"라는 말에 속아 커밋이 사라진다고 생각하지 마세요.
- `revert` 실행 시 에디터가 열리면 당황하지 마세요. 커밋 메시지를 확인하고 `:wq` (vim) 또는 저장 후 닫기를 하면 됩니다.
- 에디터가 익숙하지 않다면 `git revert --no-edit <hash>`로 기본 메시지를 자동 사용할 수 있습니다.
- `git push -f`는 **절대 main/master 브랜치에서 하지 마세요.** 이 실습에서는 개인 브랜치에서만 체험합니다.

## 실무에서의 활용

| 상황                              | 추천 방법                                           |
| --------------------------------- | --------------------------------------------------- |
| main에 버그 커밋을 push함         | `git revert`                                        |
| 내 feature 브랜치에서 실수한 커밋 | `git reset --hard` + `git push -f`                  |
| PR에 올린 커밋을 수정하고 싶음    | `git commit --amend` + `git push -f` (Scenario 4.1) |
| 여러 커밋을 합치고 싶음           | `git rebase -i` (Scenario 1.2)                      |

## 최종 기대 효과

이 시나리오를 마치면:

- "아, main에 잘못 push했다!" 상황에서 **당황하지 않고 `revert`를 사용**할 수 있습니다
- `reset`과 `revert`의 차이를 **다른 사람에게 설명**할 수 있습니다
- 공유 브랜치 vs 개인 브랜치에서의 **올바른 전략을 선택**할 수 있습니다

## 심화 도전 과제

1. **연속 revert**: 잘못된 커밋이 2개 연속인 경우, `git revert HEAD~2..HEAD`로 범위 revert를 시도해보세요.
2. **revert의 revert**: revert 커밋을 다시 revert하면 어떻게 될까요? 원래 변경사항이 복원되는지 확인해보세요.
