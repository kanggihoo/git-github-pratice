# Scenario 1.1 실습 가이드: 이미 푸시된 잘못된 커밋 수습하기

> **상황**: 당신은 `main` 브랜치에 보안 버그가 있는 인증 코드를 실수로 push해버렸습니다.
> 팀원들은 이미 pull을 받아 작업 중입니다. 안전하게 되돌려야 합니다!

---

## 🎯 미션 0: 실습 환경 구성

### 📋 해야 할 일

setup-script.sh를 실행하여 실습 환경을 구성하세요.

### 실행 방법

```bash
# practice 폴더 안에서
bash ../setup-script.sh
```

### ✅ 확인 방법

- `git log --oneline` 실행 시 4개의 커밋이 보이나요?
- 마지막 커밋이 "feat: 사용자 인증 기능 추가"인가요?
- `auth.js` 파일이 존재하나요?

## 🎯 미션 1: 현재 상태 파악하기

### 📋 해야 할 일

1. 현재 커밋 히스토리를 확인하세요.
2. 마지막 커밋에서 어떤 파일이 변경되었는지 확인하세요.
3. `auth.js`를 열어서 어떤 버그가 있는지 직접 확인하세요.

### 💡 힌트

- 커밋 히스토리를 한 줄로 보기: `git log --_______`
- 특정 커밋의 변경 내용(파일 목록)을 보기: `git show ______ --stat`
  - 가장 최근 커밋을 가리키는 키워드: `H___`
- 파일 내용 확인: `cat auth.js`

### ✅ 확인 방법

- 4개의 커밋이 시간순으로 보이나요?
- 마지막 커밋에서 `auth.js`와 `index.html` 2개 파일이 변경된 것을 확인했나요?
- auth.js에서 버그 3개를 찾았나요? (주석에 힌트가 있습니다)

## 🎯 미션 2: git revert로 안전하게 되돌리기 (핵심!)

### 📋 해야 할 일

`git revert`를 사용하여 마지막 커밋의 변경사항을 되돌리는 **새로운 커밋**을 만드세요.

### 💡 힌트

- revert 명령어: `git ______ <되돌릴-커밋>`
- 가장 최근 커밋을 가리키는 키워드: `H___`
- 에디터가 열리면 커밋 메시지를 확인하고 저장/닫기 하세요
  - vim: `:wq` 입력 후 Enter
  - nano: `Ctrl+X` → `Y` → Enter
- 에디터 없이 기본 메시지를 사용하려면: `--no-____` 옵션

### ✅ 확인 방법

- `git log --oneline`에서 커밋이 **5개**로 늘어났나요?
- 새 커밋 메시지가 `Revert "feat: 사용자 인증 기능 추가"`인가요?
- `auth.js` 파일이 **삭제**되었나요? (`ls`로 확인)
- `index.html`이 원래 상태로 돌아왔나요?

## 🎯 미션 3: revert 결과를 원격에 push

### 📋 해야 할 일

revert 커밋을 원격 저장소에 push하세요.

### 💡 힌트

- 일반적인 push 명령어를 사용하면 됩니다: `git _____ origin main`
- force push가 필요할까요? → 아닙니다! revert는 새 커밋을 **추가**한 것이므로 일반 push로 충분합니다.

### ✅ 확인 방법

- push가 에러 없이 성공했나요?
- `git log --oneline origin/main`으로 원격 히스토리를 확인했을 때 revert 커밋이 보이나요?

---

## 🎯 미션 4: (심화) 개인 브랜치에서 reset + force push 체험

### 📋 해야 할 일

1. revert 직전 상태로 새 브랜치를 만드세요 (revert 커밋의 부모 커밋에서 분기)
2. 그 브랜치에서 `git reset`으로 마지막 커밋을 **완전히 제거**하세요
3. force push로 원격에 반영하세요
4. main 브랜치로 돌아와서 두 방식의 히스토리를 비교하세요

### 💡 힌트

- 특정 커밋에서 새 브랜치 만들기: `git checkout -b <브랜치명> <커밋-기준>`
  - revert 커밋의 바로 이전 커밋: `HEAD~_` (숫자 하나)
- reset으로 마지막 커밋 제거: `git _____ --____ HEAD~1`
  - `--hard`는 워킹 디렉토리까지 모두 되돌립니다
  - `--soft`는 커밋만 취소하고 변경사항은 스테이징에 남깁니다
- force push: `git push -_ origin <브랜치명>`
- 브랜치 전환: `git ________ main`

### ✅ 확인 방법

- main 브랜치: 커밋 5개 (revert 포함)
- 새 브랜치: 커밋 3개 (버그 커밋이 히스토리에서 사라짐)
- 두 브랜치의 최종 파일 상태는 같지만, 히스토리는 다른 것을 확인했나요?
