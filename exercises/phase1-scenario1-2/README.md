# Phase 1 - Scenario 1.2: 커밋 메시지 수정 및 쪼개기/합치기 (Interactive Rebase)

## 시나리오 정보

| 항목 | 내용 |

| **Scenario** | 1.2 |

## 상황 설명

> 당신은 새 기능을 개발하며 5개의 커밋을 만들었습니다. PR을 올리려고 히스토리를 보니... `wip`, `fix typo`, `진짜 수정`, `아 또 수정` 등 보기 민망한 커밋들이 나열되어 있습니다. 이대로 PR을 올리면 리뷰어가 한숨을 쉴 것입니다.

이 시나리오를 선택한 이유: **깔끔한 커밋 히스토리**는 코드 리뷰 품질과 프로젝트 유지보수에 직결됩니다. Interactive Rebase는 커밋을 합치고, 메시지를 수정하고, 순서를 바꾸는 **가장 강력한 도구**입니다.

## 학습 목표

1. `git rebase -i`로 인터랙티브 리베이스를 시작하는 방법을 배운다
2. `squash`로 여러 커밋을 하나로 합치는 방법을 익힌다
3. `reword`로 커밋 메시지만 수정하는 방법을 배운다
4. `edit`로 커밋 내용을 수정하는 방법을 이해한다
5. 리베이스 중 문제가 생겼을 때 `--abort`로 안전하게 취소하는 법을 배운다

## 사전 요구사항

- Git 기본 명령어 사용 경험
- Scenario 1.1 완료 권장 (필수는 아님)
- 터미널 텍스트 에디터(vim 또는 nano) 기본 조작

## 핵심 학습 개념

### 왜 커밋 히스토리를 정리해야 할까?

```
❌ 나쁜 커밋 히스토리              ✅ 좋은 커밋 히스토리
──────────────────────            ──────────────────────
wip                               feat: 장바구니 기능 구현
fix                               feat: 결제 시스템 연동
진짜 수정
아 또 수정
이번엔 진짜 됨
```

좋은 히스토리는:
- 코드 리뷰가 쉬워집니다
- `git blame`으로 변경 이유를 빠르게 파악할 수 있습니다
- 문제가 생겼을 때 `git revert`로 특정 기능만 정확히 되돌릴 수 있습니다

### Interactive Rebase 동작 원리

```
[실행 전] HEAD에서 N개 전까지의 커밋을 대상으로 선택

git rebase -i HEAD~5
                │
                ▼
┌─────────────────────────────────────────────┐
│  에디터가 열리며 커밋 목록이 표시됨           │
│                                             │
│  pick abc1234 init: 프로젝트 초기 설정       │  ← 가장 오래된 커밋
│  pick def5678 wip                           │
│  pick ghi9012 fix typo                      │
│  pick jkl3456 진짜 수정                      │
│  pick mno7890 아 진짜 이번엔 진짜            │  ← 가장 최근 커밋
│                                             │
│  # 사용 가능한 명령어:                       │
│  # p, pick = 커밋 유지                       │
│  # r, reword = 커밋 유지, 메시지만 수정       │
│  # s, squash = 이전 커밋에 합치기             │
│  # f, fixup = squash + 메시지 버리기          │
│  # d, drop = 커밋 삭제                       │
│  # e, edit = 커밋에서 멈춤 (수정 가능)        │
└─────────────────────────────────────────────┘
                │
        명령어를 수정하고 저장
                │
                ▼
┌─────────────────────────────────────────────┐
│  Git이 커밋을 순서대로 다시 적용              │
│  squash/reword 등의 명령에 따라 처리          │
└─────────────────────────────────────────────┘
```

### 주요 명령어 요약

| 명령어 | 약어 | 동작 |
|--------|:---:|------|
| `pick` | p | 커밋을 그대로 유지 |
| `reword` | r | 커밋 유지, 메시지만 수정 |
| `squash` | s | 바로 위 커밋에 합치기 (메시지 편집 가능) |
| `fixup` | f | 바로 위 커밋에 합치기 (메시지 버리기) |
| `edit` | e | 커밋에서 일시 정지 (내용 수정 가능) |
| `drop` | d | 커밋 삭제 |

## 추천 실습 순서

1. **미션 1**: 현재 히스토리와 각 커밋의 변경 내용 파악
2. **미션 2**: `git rebase -i`로 커밋 합치기 (squash)
3. **미션 3**: 합쳐진 커밋의 메시지를 의미 있게 수정
4. **미션 4**: (심화) reword, edit 명령어 체험

## 교육자의 팁

- **에디터에서 당황하지 마세요!** 처음 열리는 화면은 커밋 목록입니다. `pick`을 `squash`로 바꾸기만 하면 됩니다.
- **vim 기본 조작**: `i`로 편집 모드 → 수정 → `Esc` → `:wq`로 저장 종료
- **nano 기본 조작**: 바로 수정 가능 → `Ctrl+X` → `Y` → Enter로 저장 종료
- 에디터를 바꾸고 싶다면: `git config --global core.editor "nano"` (또는 "code --wait")
- **실수했으면**: `git rebase --abort`로 언제든 리베이스 시작 전 상태로 돌아갈 수 있습니다!
- squash는 반드시 **위에 pick 커밋이 있어야** 합니다. 첫 번째 줄을 squash로 바꾸면 에러!

## 실무에서의 활용

- **PR 올리기 전**: 지저분한 커밋을 정리하여 리뷰어가 이해하기 쉬운 히스토리를 만듦
- **feature 브랜치 정리**: main에 merge하기 전 불필요한 커밋을 합침
- **커밋 메시지 수정**: Conventional Commits 형식(`feat:`, `fix:` 등)에 맞게 메시지 통일
- **잘못된 커밋 분리**: 하나의 큰 커밋을 `edit`으로 멈추고 여러 개로 쪼갬

## 최종 기대 효과

이 시나리오를 마치면:
- PR 올리기 전 **커밋 히스토리를 깔끔하게 정리**할 수 있습니다
- `squash`, `reword`, `edit` 등 **리베이스 명령어를 자유롭게 사용**할 수 있습니다
- 리베이스 중 문제가 생겨도 **당황하지 않고 `--abort`로 복구**할 수 있습니다

## 심화 도전 과제

1. **커밋 순서 바꾸기**: 에디터에서 커밋 줄의 순서를 바꿔보세요. Git이 커밋을 새 순서대로 재적용합니다.
2. **커밋 쪼개기**: `edit`으로 멈춘 후, `git reset HEAD~1`로 변경사항을 언스테이징하고, 여러 커밋으로 나눠보세요.
